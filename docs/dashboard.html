Dashboard de Ventas

Fecha de publicación

11 de marzo de 2026

Resumén:

El objetivo de este proyecto es visualizar la distribución de productos en las sucursales de CT International. Actualmente, la empresa gestiona el aspecto logístico mediante archivos de Excel. Se busca optimizar esta tarea mediante una herramienta intuitiva que permita visualizar la información de manera más eficiente. El éxito del proyecto se evaluará en función de la capacidad del dashboard para proporcionar información valiosa a los usuarios de forma diaria.

KPIs

Objetivo: Tener a la mano valores claves en la toma de decisiones relacionadas al surtimiento de productos. Los valores que consideramos útiles fueron las unidades vendidas,costo total, la ganancia total del elemento seleccionado.

Datos utilizados: De la información de facturas de venta, se utilizó el código de producto,fecha,clave de sucursal,precio de venta, costo por unidad y la cantidad de unidades vendidas.

Transformaciones: Se calcula el ingreso obtenido por venta para despues cálcular el total de unidades vendidas, la ganancia y el costo total. Estos cálculos se realizan a nivel sucursal y nivel global para tener comparación . Para calcular el ritmo de crecimiento de cada KPI, se calcula el valor de este mismo 30 días previo y se utiliza la expresión simple

\[ \text{Ritmo de crecimiento} = \frac{\text{Valor de periodo actual}-\text{Valor de periodo anterior}}{\text{Valor de periodo anterior}} * 100 % \]

Comparación de prioridades

Objetivo: Visualizar productos con prioridad junto con la cantidad de ventas que aportaron durante el periodo. El criterio de prioridad que se utiliza para asignarle magnitud de prioridad a cada producto es en base cuanto valor aporta al valor total del més. El criterio de valor es en base el costo del producto, ya que por lo general se mantiene fijo.

Datos utilizados:De la información de facturas de venta, se utilizó el código de producto,fecha,clave de sucursal, y la cantidad de unidades vendidas.

Transformaciones: Primero se agrupa la información por elemento y sucursal para calcular las unidades vendidas. Después, calcula el valor aportado por cada producto multiplicando el costo por las ventas totales. Luego, se ordena la información en base el valor y se hace una suma acumulativa total para poder acertar cuanto aportó cada producto al valor adquirido total del periodo especificado. Si el producto aportó al 80% del valor total, se considera de alta prioridad, 15% media y 5% baja.

Interpretabilidad: La idea principal es identificar los productos que representan el mayor porcentaje del ingreso total de la sucursal seleccionada. Al visualizar estas prioridades, se busca apoyar la toma de decisiones al momento de surtir productos de manera intuitiv.(Sensible a ventas atípicas)

Preparación de datos

Carga

Las fuentes de datos principales de cual se extraen los datos crudos vienen siendo :

Fuente Descripción Información extraída
Data WareHouse (SQL Microsoft Server) Base de datos en cual se almacena información básica de productos y de todas las facturas realizadas. Información relacionada con facturas de venta y con costo de productos.
CT Online (SQLite) Base de datos en cual se almacena información básica de productos Categoría al cual pertenece cada producto.
CT API (mongodb) Base de datos variada en cual se almacena información de facturas,paquetería, existencia de productos etc. Información actualizada de almacenamiento y existencia de productos.
CT Historico (mongodb) Base de datos en cual se almacena histórica de almacenamiento y existencia de productos Información histórica de almacenamiento y existencia de productos.

Procesamiento

Primero, se revisa si la tabla local de facturas esta actualizada al día. En caso que no lo esté, se extraen solo las filas que hacen falta y se agregan a la tabla. Una vez que se haya actualizado la información de cada tabla, se pasan por filtros. En caso de los datos de facturas, se tienen que filtrar para tener solo facturas de venta y de productos físicos. Es decir, se omiten suscripciones u otros servicios. Del folio de la factura se extrae la homoclave de la sucursal en cual se realiza la venta para lograr juntar la información de facturas con la información de almacenamiento por sucursal.

Ingesta de datos históricos

Se identificó una limitación importante: no existía un registro histórico de las existencias de productos. Solo se registra la cantidad disponible en un momento determinado, lo que impide trazar la evolución de inventarios a lo largo del tiempo. Esta información es clave tanto para el dashboard como para el proyecto en general de surtimiento. Por ello, se creó la base de datos CT Historico que almacene este historial, permitiendo un análisis más completo y estratégico. Para optimizar espacio de almacenamiento, se quieren guardar y/o actualizar solo cambios de valores, y no documentos completos de manera diaria. El script de ingesta/ctualización de datos históricos ya esta implementada y se esta programada para ser ejecutada diariamente a las 6:00 a.m (UTC-7,tiempo de Hermosillo). Validación de datos

Para obtener valor del dashboard, se requiere tener confianza en que los datos plasmados/utilizados reflejen la realidad. Por esto, es importante tener criterios de validación de estos que aseguren que los datos usados reflejan correctamente la realidad del negocio.

  • Registros no nulos : Para nuestro caso, los registros nulos no nos sirven para nada por el momento. Es por ello que los filtraremos del conjunto final de datos.

  • Formatos consistentes de columnas : Se definen en base el diccionario de datos los formatos que deben tener cada columna, cuales se utilizaran como el criterio principal de validación.

  • Suma de ventas por producto total deben ser igual a la suma de categorias total : Esta validación consiste en verificar si se hace la agrupación/separación entre productos y categorías correctamente. Para ello, se compara la suma de ventas a nivel producto con la suma de ventas a nivel categoría.

  • Precios,costos, cantidades positivas : Se revisan que columnas como price ,cost, sales_{frequency} caigan dentro de su rango lógico de valores.

  • Precio de unidad debe ser mayor a su costo : Para que nuestro conjunto de datos refleje información que tenga sentido, un requisito que se debe cumplir es que el precio de producto sea mayor a su costo.

  • Rango de fechas dentro de 2020 y actualidad A nosotros y los usuarios solo nos interesan fechas que se encuentren dentro del periodo del 2020 hasta la actualidad, ya que no hay registro de facturas fuera de este periodo.

  • Facturas únicas: Esto es para probar que los registros de facturas de venta sean únicos. El criterio que se utiliza es en base los siguientes elementos: código de producto, número de folio, código de cliente y fecha.
